## 任务3：基于 Tkinter 的实时点对点语音通信系统

本目录实现了一个带图形界面的实时语音通信系统。  
系统整体采用：

- `TCP` 作为信令通道
- `UDP` 作为语音媒体通道
- `P2P 优先、服务器中转兜底` 的传输策略
- `短时抖动缓冲 + 简单丢包补偿` 的接收播放策略

它适合用于课程设计、实验报告或答辩展示，重点体现以下几个网络设计思想：

- 控制面与数据面分离
- 实时业务优先低时延，因此媒体流使用 UDP 而不是 TCP
- 先尝试端到端直连，失败再回退到服务器中转
- 接收端不依赖重传，而是通过缓冲和插值来平滑播放

---

## 1. 功能概览

当前版本支持：

- 用户连接到同一台服务器并注册昵称
- 服务器维护在线用户列表，并同步给所有客户端
- 客户端图形界面中直接查看在线用户
- 单击在线用户填入目标昵称，双击直接发起呼叫
- 发起呼叫、接听、拒绝、挂断
- 接听后自动开启 UDP 语音发送
- 通话中暂停/恢复本地语音发送
- 文本消息收发
- P2P 直连协商
- P2P 协商失败时自动切换为服务器 UDP 中转
- 日志显示链路协商过程、最终路由、媒体首包路径、暂停发送、丢包补偿等信息

---

## 2. 设计目标

本任务的核心目标不是“做一个完整商业电话系统”，而是围绕实时语音通信，完成以下教学目标：

1. 理解实时业务为什么更适合 UDP。
2. 理解信令和媒体分离的设计方式。
3. 理解 P2P 直连和服务器中转的优缺点。
4. 理解抖动缓冲、乱序重排、丢包补偿等接收端处理逻辑。
5. 用可视化界面展示网络状态与运行过程。

---

## 3. 总体架构

系统采用“服务器协调 + 客户端直接通信”的混合架构。

### 3.1 分层思路

- `TCP 信令层`
  - 登录
  - 在线用户同步
  - 呼叫/接听/拒绝/挂断
  - 文本消息
  - P2P 成功确认
  - 中转回退通知

- `UDP 媒体层`
  - 音频帧传输
  - 探测包（probe）
  - 实时播放

### 3.2 角色划分

- `服务器`
  - 负责维护在线用户状态
  - 负责呼叫建立与信令转发
  - 负责在 P2P 未确认成功时提供 UDP 中转
  - 不负责重传媒体数据

- `客户端`
  - 负责 GUI 交互
  - 负责录音、发送、接收、播放
  - 负责发起 P2P 探测
  - 负责接收端重排、缓冲、补偿

### 3.3 当前路由策略

当前实现不再通过“固定子网掩码 + 服务器静态判定”来直接决定是否 P2P。  
而是采用更符合实际网络行为的策略：

1. 通话建立后，双方先进入 `Negotiating` 状态。
2. 客户端优先向对端 UDP 地址发送探测包与媒体流。
3. 同时，为了保证可达性，也允许媒体暂时发往服务器 UDP 中转。
4. 如果服务器收到双方都报告“已经看到对端直连 UDP”，则下发 `transport_update(mode=p2p)`。
5. 如果在协商超时时间内没有成功确认直连，则下发 `transport_update(mode=relay)`。

也就是说，系统采用的是：

`P2P first -> runtime verification -> fallback relay`

这比“按 /24 掩码算是不是同一子网”更符合真实网络环境。

## 6. 默认配置

当前默认网络和音频参数定义在 `audio_config.py` 中，主要包括：

- 默认服务器地址：`localhost`
- 默认端口：`8880`
- 采样率：`16000 Hz`
- 声道：`1`（单声道）
- 采样格式：`16-bit PCM`

客户端内部用于实时语音发送的小帧大小由 `stream_client.py` 中的：

- `MEDIA_CHUNK_SIZE = 320`

决定。  
对应 16kHz 单声道下，每帧约 `20ms` 音频，这样既能降低封包时延，也适合做短时缓冲与补偿。

### 8.2 客户端 GUI

## 9. 协议设计

### 9.1 控制消息

控制消息基于：

- `TCP`
- `JSON`
- `\n` 作为消息分隔符

当前主要控制消息类型包括：

- `login`
- `response`
- `user_list`
- `text`
- `call_invite`
- `call_accept`
- `call_reject`
- `call_hangup`
- `call_busy`
- `call_not_found`
- `call_ready`
- `transport_update`
- `direct_path_seen`
- `media_stop`

### 9.2 媒体消息

媒体消息基于：

- `UDP`
- `JSON`

当前主要媒体包类型：

- `audio_frame`
- `media_probe`

### 9.3 音频帧格式

每个音频帧至少包含：

- `stream_id`
- `sequence`
- `timestamp_ms`
- `sender`
- `target`
- `mode`
- `data`

其中：

- `sequence` 用于乱序重排与丢帧判断
- `timestamp_ms` 用于表示帧的生成时间
- `data` 为 Base64 编码后的原始 PCM 音频

## 10. 呼叫建立与链路协商流程

### 10.1 呼叫建立

1. 主叫发送 `call_invite`
2. 被叫点击接听，发送 `call_accept`
3. 服务器为本次通话生成唯一 `call_id`
4. 服务器向双方下发 `call_ready`
5. 双方进入 `Negotiating` 状态

### 10.2 协商阶段

在 `Negotiating` 状态下：

- 客户端持续向对端发送 `media_probe`
- 同时允许媒体流发往：
  - 对端 UDP 地址
  - 服务器 UDP 中转地址

这样做的原因是：

- 如果直连能成功，可以尽快确认 P2P
- 如果直连不通，服务器中转也能保证通话不中断

### 10.3 直连确认

如果客户端收到了来自对端真实 UDP 地址的 `probe` 或音频包，就向服务器发送：

- `direct_path_seen`

当服务器确认：

- `caller` 已看到对端直连包
- `callee` 已看到对端直连包

就会广播：

- `transport_update(mode=p2p)`

之后双方只保留直连路径。

### 10.4 中转回退

如果在协商超时内，服务器没有收到双方都确认的直连消息，则广播：

- `transport_update(mode=relay)`

之后双方统一只使用服务器 UDP 中转。

## 11. 在线用户同步机制

在线用户列表由服务器维护，是系统中的“唯一权威来源”。

### 11.1 服务器维护方式

服务器内部保存：

- `nickname -> connection id` 的映射
- 客户端 TCP 连接信息
- 客户端 UDP 媒体端口信息

### 11.2 同步时机

服务器会在以下时机自动更新并广播在线列表：

- 用户登录成功后
- 用户断开连接后

### 11.3 首次连接快照

为了避免“用户 A 先连，用户 B 后连时，B 需要等下一次变化才能看到 A 在线”的问题，  
服务器在 `login` 成功响应中就会附带一份当前在线用户快照。

这意味着：

- 新连接的客户端在刚登录成功后，就能立刻知道有哪些用户已经在线

## 12. 接收端播放设计

### 12.1 为什么需要抖动缓冲

UDP 不保证：

- 按序到达
- 不丢包
- 固定时延

因此直接“收到就播”会出现：

- 抖动
- 爆音
- 播放断裂

### 12.2 当前实现

客户端收到 UDP 音频帧后，会：

1. 按 `sequence` 放入缓冲字典
2. 等待一个很短的播放延迟窗口
3. 按顺序取出并播放

这样可以吸收一部分网络抖动。

### 12.3 丢包补偿

当前不采用重传，原因是：

- 实时语音更怕“太晚到”，而不是“少一帧”

所以接收端使用简单的 Packet Loss Concealment（PLC）：

- 如果上一帧和下一帧都可用：
  - 用前后帧插值生成补偿帧
- 如果只有上一帧：
  - 重复上一帧并做轻微衰减
- 如果只有下一帧：
  - 临时借用下一帧
- 如果都没有：
  - 生成静音帧

这种策略虽然简单，但在实验环境中已经足以体现实时语音系统的设计思路。

---

## 13. 暂停语音发送的处理方式

通话过程中点击“暂停语音发送”时：

- 发送端停止本地麦克风采集
- 发送端通过 TCP 信令发送 `media_stop`
- 对端收到后清空当前接收缓冲，停止继续把后续序号当成“持续丢包”

这样可以避免以前那种：

- 一旦停止发送
- 接收端仍然持续推断“后续所有帧都丢失”
- 日志不断刷 `Missing UDP frame ...`

的问题。

## 14. 主要状态机

客户端通话状态定义为：

- `IDLE`
- `CALLING`
- `RINGING`
- `CONNECTING`
- `IN_CALL`
- `ENDED`

其中：

- `CALLING`：已发起呼叫，等待对方接听
- `RINGING`：收到来电，等待用户接听/拒绝
- `CONNECTING`：对方已接听，等待协商消息
- `IN_CALL`：通话中，且可能处于：
  - `negotiating`
  - `p2p`
  - `relay`

## 15. 关键实现特点

### 15.1 采用控制面/数据面分离

- 控制面走 TCP，保证状态消息可靠
- 数据面走 UDP，优先实时性

### 15.2 协商期双路径发送

在 `Negotiating` 阶段，客户端会把音频同时发给：

- 对端
- 服务器

接收端通过序号去重，所以即使两路都到达，也不会被当成两份不同的音频重复播放。

### 15.3 在线用户以服务器为准

客户端不自己猜测谁在线，而是完全以服务器下发的列表为准。

### 15.4 GUI 与核心逻辑解耦

- GUI 只负责展示与交互
- `stream_client.py` / `stream_server.py` 才是核心网络逻辑

这种结构更利于后续：

- 替换界面
- 增加 CLI 客户端
- 编写自动化测试

## 16. 当前限制

当前实现是课程设计取向，仍有一些限制：

- 没有做 NAT 打洞，P2P 仅依赖双方能直接互通 UDP
- 探测策略较简单，没有 STUN / TURN / ICE
- 音频数据使用 JSON + Base64，编码效率不如原始二进制帧
- 没有做语音编解码压缩，直接传 PCM，带宽占用较高
- 丢包补偿是简单插值，不是专业级 PLC
- 没有鉴权、加密、账号系统
- 没有历史消息持久化

## 18. 建议答辩表述

如果需要做答辩，可以用下面这段思路概括：

> 本系统采用 TCP 负责信令、UDP 负责实时音频传输。  
> 在通话建立后，系统先尝试 P2P UDP 直连，并通过运行期探测确认直连是否真实可用；如果直连失败，则自动回退到服务器 UDP 中转。  
> 接收端采用短时抖动缓冲、按序播放以及简单丢包补偿机制，在不依赖重传的前提下尽量降低播放中断和听感突变。  
> GUI 层则对在线用户、呼叫流程、发送暂停与运行日志进行了可视化展示。
