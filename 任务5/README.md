# 任务5 - 多方语音会议系统

这是在任务4基础上扩展出的多人语音会议系统。当前版本的控制面仍然使用 `TCP + JSON`，但音频数据面已经从“服务器逐个转发的模拟组播”改成了“客户端直接收发的 `UDP + IP Multicast`”。

## 当前实现要点

- 服务端只负责登录、联系人、邀请、入会/退会、解散会议等信令控制。
- 每个聊天室创建时，服务端会分配一个组播地址和端口，并在创建/加入响应里下发给客户端。
- 客户端通过 UDP 向该组播组发送音频，所有已加入该组播组的成员直接接收，不再经过服务端音频中转。
- 每个音频包都携带一份“上一帧”的冗余副本，用于在单帧丢包时优先本地恢复。
- 接收端使用小型抖动缓冲区延迟播放；若冗余恢复失败，则对缺失帧做插值/衰减补偿，尽量避免爆音和卡顿。
- 设计目标是“低时延优先”，因此针对丢包默认不做重传。

## 文件说明

| 文件 | 说明 |
|------|------|
| `conference_protocol.py` | 会议控制协议定义与编解码 |
| `conference_server.py` | 会议服务端，负责 TCP 信令与组播地址分配 |
| `conference_client.py` | 会议客户端，负责控制逻辑、组播音频收发、缓冲播放 |
| `multicast_audio.py` | IP 组播音频传输辅助模块，包含包格式、冗余恢复、抖动缓冲与插值补偿 |
| `client_gui.py` | 客户端 GUI |
| `server_gui.py` | 服务端 GUI |
| `test_core.py` | 控制面集成测试 |
| `test_multicast_audio.py` | 音频传输层测试 |

## 运行方式

### 1. 启动服务端

```bash
cd 任务5
python server_gui_start.py
```

或：

```bash
cd 任务5
python server_start.py
```

### 2. 启动客户端

```bash
cd 任务5
python client_gui_start.py
```

### 3. 基本流程

1. 启动服务端。
2. 启动多个客户端，并分别登录不同用户名。
3. 由一名用户创建聊天室。
4. 在聊天室中邀请其他在线用户加入。
5. 被邀请用户接受后，客户端会加入同一个 IP 组播组。
6. 会内所有成员通过 UDP 组播直接交换音频。

## 测试

控制面测试：

```bash
python test_multicast_audio.py
```

## 和旧实现的区别

旧版“UDP 模式”仍然是应用层组播：服务端收包后逐个 `sendto` 转发给房间内成员。

当前版本改为网络层 IP 组播后：

- 服务端不再承载房间音频转发压力。
- 应用层协议固定为 UDP。
- 丢包恢复主要依赖编码冗余、缓冲和插值，而不是重传。
